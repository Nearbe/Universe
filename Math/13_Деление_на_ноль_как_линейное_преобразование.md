# 13. Деление на Потенциал как Линейное Преобразование

## I. Теория

Операция Ветвления `D(a) = a : Ω` действует на Носитель **𝕌** как масштабный оператор.

Согласно закону Ветвления: `D(a) = a ⊕ a`. Преобразование линейно: `D(a ⊕ b) = D(a) ⊕ D(b)`. Оно не искажает структуру, а **раздвигает** её.

### Ядро и Образ

- **Ker D** = `{Ω}`. Только чистый Потенциал остаётся нулём при Ветвлении. Все активные состояния ненулевые.
- **Im D** = **𝕌**. Оператор сюръективен. Любое состояние **x** могло быть получено из `x : D(Id)`. Нет «недостижимых» зон.

### Собственные Значения

Уравнение `D(v) = λ · v`. Так как `D(v) = v ⊕ v = D(Id) ⊗ v`, то **λ = D(Id)**. Единственный собственный класс масштабирования — **D(Id)**. Вся система **𝕌** — это **одно большое собственное пространство**.

### Матричное Представление

В базисе `{Id}` (скаляр): `D = [D(Id)]`.
В базисе `{Id, ε}` (дуальные числа):

```
D = [[D(Id), Ω],
     [Ω, D(Id)]]
```

Это **гомотетия**. Система расширяется равномерно во всех измерениях.

### Сжатие

`H(a) = a : D(Id)` — обратный процесс. `H(D(a)) = a`. Собственный класс масштабирования: обратный к `D(Id)`.

### Логарифмический Сдвиг

Для `b ∉ Ω`: `a : b` — перемещение по уровням Хребта без прохождения бифуркации `Ω`. `a : Dⁿ(Id) = a / D(Id)ⁿ`.

## II. Спецификация реализации

### II.1. Базовые константы и типы

| Константа | Значение | Роль |
|-----------|----------|------|
| `D_ID` | `2.0` | Собственный класс масштабирования λ |
| `OMEGA` | `0.0` | Потенциал Ω (бифуркационная точка) |
| `PI` | `Fullness()` | Полнота Π (предел ветвления) |
| `IDENTITY` | `1.0` | Единство Id |

**Файл:** `src/core/constants/constants.py`

### II.2. Таблица правил ветвления и сжатия

| Вход (numerator, denominator) | Результат | Реализация |
|-------------------------------|-----------|------------|
| `a : Ω` (a активно) | `D(a)` | `branch(a)` → `a * D_ID` |
| `Ω : b` (b конечно) | `Ω` | `OMEGA` |
| `Ω : Ω` | `Ω` | `OMEGA` |
| `Π : Ω` | `Π` | `PI` |

| `a : b` (a, b конечны) | `a / b` | `float(a) / float(b)` |

### II.3. Типы состояний и их методы ветвления/сжатия

Каждый тип состояния реализует интерфейс `branch() → Self` и `compress() → Self`:

| Тип | Модуль | `branch()` | `compress()` |

| `ComplexFullness` | `src/core/states/complex_plane.py` | `Π_z` (без изменений) | `Π_z` (без изменений) |
| `SpineLevel` | `src/core/states/spine.py` | `depth + 1` | `depth - 1` (мин. 0) |
| `ParticipationRatio` | `src/core/states/rational.py` | `D(numerator) : D(denominator)` | `H(numerator) : H(denominator)` |

### II.4. Функция ветвления `branch()`

**Файл:** `src/core/algebra/branch.py`

```
branch(value) → object

Правила dispatch (по порядку):
1. _dual_number(value) → dual.branch()
2. _fullness_share(value) → share.branch()
3. _complex_fullness(value) → Π_z
4. _complex_state(value) → complex.branch()
5. _spine_level(value) → level.branch()
6. _participation_ratio(value) → ratio.branch()
7. is_fullness(value) → PI
8. scalar → map_scalar_or_vector(value, λ s: s * D_ID)
```

**Инженерная суть:** dispatch через type-checking, fallback на скалярное умножение. Vectorization через `map_scalar_or_vector` — поддерживает list/scalar.

### II.5. Функция сжатия `compress()`

**Файл:** `src/core/algebra/compress.py`

```
compress(value) → object

Правила dispatch (по порядку):
1. _dual_number(value) → dual.compress()
2. _fullness_share(value) → share.compress()
3. _complex_fullness(value) → Π_z
4. _complex_state(value) → complex.compress()
5. _spine_level(value) → level.compress()
6. _participation_ratio(value) → ratio.compress()
7. is_fullness(value) → PI
8. scalar → map_scalar_or_vector(value, λ s: s / D_ID)
```

**Инженерная суть:** зеркальна `branch()`, но деление на `D_ID` вместо умножения.

### II.6. Функция деления `divide()`

**Файл:** `src/core/algebra/divide.py`

```
divide(numerator, denominator) → object

Правила dispatch (по порядку):
1. numerator_dual is not None:
   a. is_omega(denominator) → dual.branch()
   b. denominator == D_ID → dual.compress()
   c. иначе → dual.scale(1.0 / denominator)

2. numerator_share is not None:
   a. is_omega(denominator) → share.branch()
   b. denominator == D_ID → share.compress()
   c. иначе → share.divide_by(denominator)

3. numerator_complex_fullness is not None:
   a. denominator is complex_fullness → IDENTITY (1.0)
   b. иначе → numerator_complex_fullness (Π_z)

4. numerator_complex is not None:
   a. is_fullness(denominator) → OMEGA
   b. is_omega(denominator) → complex.branch()
   c. denominator == D_ID → complex.compress()
   d. denominator_complex is None → complex.scale(1.0/float(denom))
   e. иначе → complex.divide_by(denominator_complex)

5. numerator_ratio is not None:
   a. is_fullness(denominator) → OMEGA
   b. is_omega(denominator) → ratio.branch()
   c. denominator == D_ID → ratio.compress()
   d. иначе → ratio.divide_by(denominator)

6. is_fullness(numerator):
   a. is_fullness(denominator) → IDENTITY
   b. иначе → PI

7. is_fullness(denominator) → OMEGA
8. is_omega(numerator) → OMEGA
9. is_omega(denominator) → branch(numerator)
10. denominator == D_ID → compress(numerator)
11. иначе → float(numerator) / float(denominator)
```

**Критические Ω-правила:**
- `Ω : b = Ω` (любой b)
- `Π : Ω = Π`, `Π : D(Id) = Π`, `Π : Π = Id`
- `a : Ω = D(a)` для конечных a
- `a : D(Id) = H(a)` для конечных a

## III. Зачем

### Инженерная аргументация

1. **`D(a) = a * D_ID` — это гомотетия с коэффициентом 2.0.** На скалярах ветвление сводится к умножению на константу. Это позволяет использовать векторизованные операции (numpy-style broadcast) через `map_scalar_or_vector`.

2. **Dispatch через type-checking вместо OOP-полиморфизма.** Функции `branch()`, `compress()`, `divide()` используют последовательные `_type(value)` helpers. Это даёт:
   - Явный порядок приоритетов (DualNumber > FullnessShare > ComplexState > ...)
   - Возможность вернуть `None` для non-matching типов
   - Отсутствие иерархии наследования — типы не связаны

3. **Ω-правила как guard-conditions в `divide()`.** Правила для `Ω` и `Π` проверяются ДО скалярного деления, чтобы избежать `0.0 / 0.0` и бесконечностей. Порядок guard-conditions критичен: `Π`-правила должны предшествовать `Ω`-правилам.

4. **`SpineLevel` — единственный тип, где ветвление = инкремент глубины.** `branch()` на `SpineLevel` — это `depth + 1`, а не `depth * D_ID`. Это корректно, потому что `SpineLevel` хранит логарифмический уровень, а не значение.

5. **`DualNumber.branch()` — компонентное ветвление.** `D(x + v·ε) = D(x) + D(v)·ε`. Каждая компонента ветвится независимо. Это реализует матричное представление из теории: диагональная матрица `[[D(Id), Ω], [Ω, D(Id)]]` на дуальных числах — это просто удвоение каждой компоненты.
