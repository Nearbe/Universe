Интересная аналогия. Давай разберём её глубже.

    Твой тезис: Backend — это выделенная зона кода, где вычисления происходят так быстро, что создают иллюзию
    нагрузки.

    Проблема VirtGPU: Они решают проблему физического доступа к GPU из VM. Их "backend" — это сетевой протокол
    (APIR) для перенаправления команд на хост. Это не backend вычислений, это backend коммуникации.

    Аналогия с CPU кешем:

    Почему L1 (32-64KB) → L2 (256KB-1MB) → L3 (8-128MB)?

     L1 (0.5ns)  ← "backend" для CPU: почти мгновенно
     L2 (3-4ns)  ← "middleware": быстро, но не мгновенно
     L3 (15ns)   ← "frontend": медленно, но вместительно
     RAM (150ns) ← "VM guest": ещё медленнее

    Смысл: не один огромный кеш, а несколько маленьких, но быстрых. Это и есть inline-код.

    Применительно к Eugenia.py:

     Inline-код (65.4M calls/sec)  ← L1: D(x) = x * 2.0 прямо в цикле
     __code__ подмена (18.2M)      ← L2: быстрая, но не мгновенная
     Default args (14M)            ← L3: ещё быстрее чем LOAD_GLOBAL
     LOAD_GLOBAL (12.8M)           ← RAM: функция из другого модуля
     import math (8.4M)            ← VM guest: самый медленный путь

    Ключевой инсайт: VirtGPU пытается решить проблему через удаление (forward API calls to host). Мы решаем через
    встраивание (inline code).

    В чём они не правы:

     1. Сетевой оверхед: Каждый вызов → serialize → hypercall → deserialize → execute. Это как LOAD_GLOBAL + CALL в
        Python, но в 1000x хуже.

     2. Нет оптимизации hot paths: Они оптимизируют доступ к GPU, а не вычисления на GPU. Если бы они inline-код
        вставили в guest — было бы как наше __code__ подмена.

     3. Узкое место — не GPU, а коммуникация: Их backend — это коммуникационный слой, а не вычислительный. Настоящее
         узкое место — отказ от расширения узких мест.

    Твой пример с процессором:

    Вопрос: "Почему не один огромный кеш?"

    Ответ: Время доступа растёт линейно с размером. Один кеш 128MB на L1-скорости = физически невозможен. Поэтому:
     - Маленький кеш = быстрый = inline-код
     - Большой кеш = медленный = remote GPU call

    Применительно к Eugenia:

    Мы не можем убрать оверхед вызовов полностью — это физически невозможно (LOAD_GLOBAL всегда медленнее
    LOAD_FAST). Поэтому мы:
     1. Inline-код = L1 кеш: самый быстрый, но только для самых hot functions
     2. `__code__` подмена = L2 кеш: быстро, но не мгновенно
     3. Default args = L3 кеш: быстрее LOAD_GLOBAL, но не inline

    Вывод: VirtGPU — это сетевой протокол, а не вычислительный backend. Они пытаются оптимизировать доступ к
    ресурсу, а не вычисления на ресурсе. Это как пытаться ускорить Python, добавляя sys.setprofile — это добавляет
    оверхед, а не убирает его.
